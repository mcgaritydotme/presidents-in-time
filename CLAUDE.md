# Claude-specific guidance for presidents-in-time

The full workflow is documented in README.md. This file contains behavioral rules specific to Claude Code.

## Always present candidates before acting

For every four-year block, present 1–3 candidates with images/links and **wait for the user to pick one** before downloading, writing JSON, or generating images. This applies even when only one good candidate exists. Never skip straight to the download step.

## All artwork types are equal

Do not bias toward paintings. Engravings, lithographs, daguerreotypes, photographs, sculptures, and other media are all valid — present them without apology or qualification. A period-accurate print beats an off-period oil painting. See README.md Step 2 for the full list of accepted types.

## After the user picks

Download immediately, generate popover/thumbnail, and update `presidents.json` — all without waiting to be asked. Then report what was saved and prompt for the next block.

## Batch mode

The rules above are the default, interactive, per-block workflow. When the user explicitly asks for a **batch run** on a president (e.g. "run all periods for Harrison," "batch this one"), switch to the process in README.md's Batch Mode section instead. Behavioral specifics Claude must follow:

- **Bias toward 3 candidates per period.** Actively try to find three quality candidates per four-year block; returning fewer is acceptable only once you've genuinely exhausted the search, not as a default.
- **No per-period pause.** Move from period to period without stopping for approval. Collect silently; do not narrate each block as you go.
- **Capture full metadata per candidate**, every time, even for candidates that don't end up chosen: artist, date, medium, **dimensions in inches**, institution, image link, and a one-line note on historical significance (official portrait, painted by a friend, campaign artifact, etc.).
- **Domain handling during collection:**
  - If the candidate's image is hosted on a domain already in `.claude/settings.json`'s allowlist, fetch and verify it inline as part of collection — no flag, no pause.
  - If the domain is **not** on the allowlist, do not fetch it. Log it as a pending candidate with whatever metadata you can gather without fetching (artist, date, description from search snippets), and keep collecting the rest of the run. Never let an unapproved domain interrupt the batch.
  - **If a source still blocks access after being allowlisted (403, JS-only shell, bot protection), don't silently drop the lead.** Present its URL in the results so the user can visit and manually review it themselves — they often can get past blocks Claude can't. This applies any time automated access fails, not just for not-yet-approved domains.
  - **The moment the user approves a new domain, add it to `.claude/settings.json` immediately** — don't just proceed with the fetch and forget to persist it. The allowlist is the whole point; it has to actually grow each run. Write the file *before* retrying the approved fetches, not after, so the retries themselves don't prompt.
- **Prefer WebFetch over Bash curl for page scraping during collection.** The domain allowlist only governs the WebFetch tool — `Bash(curl ...)` is matched per exact command, not by domain, so it still prompts even for allowlisted domains like `ids.si.edu` or `loc.gov`. Use WebFetch to verify a candidate exists and looks right during Phase 1. Reserve `Bash curl` + PIL (for exact pixel dimensions) for Phase 5, after curation, when the user is already at the keyboard approving downloads.
  - **Exception: structured JSON APIs.** Smithsonian Open Access (`api.si.edu`), LOC (`?fo=json`), Wikimedia (`commons.wikimedia.org/w/api.php`), Met Museum (`collectionapi.metmuseum.org`), and CONTENTdm (`dmwebservices`) return JSON, not HTML — WebFetch's prose-summarizing model handles that poorly and sometimes outright fails (403s seen on item pages WebFetch couldn't reach but curl could with proper headers). `Bash curl` against these specific JSON endpoints is fine during Phase 1 collection; it's querying metadata, not bypassing the page-scraping rule above.
  - **`www.loc.gov` item pages: skip WebFetch entirely, go straight to curl.** WebFetch has 403'd on every `loc.gov/item/...` page tried this project, JSON or not — it's not worth attempting first. For any LOC item lookup, use `curl -s "https://www.loc.gov/item/<id>/?fo=json" -A "Mozilla/5.0 ..." -H "Referer: https://www.loc.gov/search/"` directly. If that 403s too (a "Just a moment..." Cloudflare challenge page instead of JSON), retry once with a more browser-like `User-Agent` before treating it as a genuine failure.
  - **`ohiomemory.org`: use the CONTENTdm JSON API, not the human-readable item page.** WebFetch against `/digital/collection/.../id/.../` returns only generic CONTENTdm platform boilerplate, no real metadata. Go straight to `curl -s "https://ohiomemory.org/digital/bl/dmwebservices/index.php?q=dmGetItemInfo/<collection>/<id>/json"` instead.
  - **This exception only avoids prompts if the rule actually matches.** `Bash` permission rules are plain prefix matches (`Bash(<literal prefix>*)`) — there's no domain-aware matching like `WebFetch(domain:X)` has. `.claude/settings.json` pre-allows these exact prefixes: `curl -s "https://api.si.edu/`, `curl -s "https://www.loc.gov/`, `curl -s "https://tile.loc.gov/`, `curl -s "https://commons.wikimedia.org/w/api.php`, `curl -s "https://ohiomemory.org/digital/bl/dmwebservices/`, `curl -s "https://collectionapi.metmuseum.org/`, `curl -sL "https://web.archive.org/`. Always invoke curl against these endpoints starting exactly `curl -s "https://<domain>/...` (or `curl -sL "https://web.archive.org/...` for that one) — no flags before the URL, no wrapping it in a loop or chaining a second command on the same line (e.g. no trailing `wc -c` or `file` on a second line — check the `-w "%{http_code}"` output or just trust the call and inspect the saved file in a later, separate call) — or the prefix match fails and it prompts anyway. New CONTENTdm domains (beyond ohiomemory.org), or any other new trusted JSON/archive endpoint, need their own prefix rule added the same way new WebFetch domains do.
  - **None of this changes the non-blocking rule above.** This guidance is about which tool to reach for *first* against domains already on the allowlist — it doesn't override the domain-handling rule two bullets up. If a call to one of these endpoints still fails (rate-limited API key, a genuinely new/unapproved domain, a permission prompt because a prefix didn't match), log it and keep collecting; resolve it at the Phase 2 domain-approval checkpoint or the end-of-run rerun, never by stopping the batch mid-run.
- **End-of-run, two separate checkpoints, never combined:**
  1. **Domain approval only.** Present the flagged-domains list (one line each: domain, what's behind it, which period it affects) and wait for the user to approve/deny each in one pass. Do not show the candidate list yet at this point.
  2. After any approved domains are fetched in a single batched rerun (parallel fetches, merge results), present the **complete** candidate list — this is the only time the full set appears, and it includes everything: originally-collected candidates plus newly unlocked ones.
- **Never use `mcp__visualize__show_widget` (or any image-embedding widget) for candidates.** Institutional image hosts (LOC, Smithsonian, Ohio Memory, Getty, etc.) block hotlinking from unrecognized referrers, so embedded `<img>` tags render as broken placeholders. Always present candidates as a plain markdown list in chat text, grouped by four-year block, with the same metadata density used for James K. Polk's run: artist, date, medium, dimensions, institution, and a one-line significance note. This applies to both the end-of-run candidate list and any other point candidates are shown.
- **Two links per candidate, every time: a direct image link and a source/object-page link.** Never present a candidate with just one. The image link is what the user actually wants to look at; the page link is the fallback when the image link is broken, slow to load, or 404s (which happens — see the NPG.99.158 case where the catalog page worked but the IDS image asset didn't exist). If a direct image URL genuinely cannot be found, say so explicitly and give the page link alone rather than silently dropping the image link.
- **Curation happens once**, against the complete candidate list, after the domain-approval rerun — never before.
- **Bulk finalization.** Once the user has picked across the whole sheet, download/resize/generate/update JSON for every selection in one pass, not one at a time.
- **Unused candidates become the placeholder pool** for any gap periods, applying the existing "most-famous-unused" rule (see README.md's Default Portrait section) — drawn from by eye, no second research pass, same finalization routine.
- **Default portrait is mandatory, not optional follow-up.** Bulk finalization is not done until the default portrait is also resolved. As soon as the block picks are finalized, go straight into researching the default-portrait candidate pool — don't pause to ask permission to start. Present the resulting options and wait for a pick before declaring the run done; the pause belongs at the pick, not at the decision to begin researching. A run isn't complete while `defaultPortraitUrl` still points at the `placehold.co` placeholder.
- **Default-portrait pool has two sources, both required.** (1) Leftover candidates from block research that didn't get picked. (2) Posthumous/retrospective depictions — created any time after the president's death — found via a dedicated search, same as if it were its own four-year block: bias toward **three** quality candidates, full metadata per candidate, before accepting fewer. Don't rely on leftovers alone; always run the posthumous-works search too, then present the combined set.
- **Citation lint pass, after the default portrait is picked and saved, before declaring the run done.** Both Harrison's and Tyler's runs needed end-of-run corrections (missing dimensions, an institution traced to the wrong place, an artist attribution LOC itself flagged as unverified caption-card data) that should have been caught earlier. For every entry just written (all block portraits plus the default), re-pull the primary record for each and check: does the artist attribution match what the institution's own record says, including any "unverified"/"attributed to"/uncertain-authorship caveats the source itself flags? Are dimensions present, and in inches? Does the named institution match the actual current holding record, not an inferred one? Fix anything that fails before reporting the run complete — this is a required step, not a follow-up to wait for the user to request.
